## 새 발견 / 함정

### 함정 ①: tsx + Node v24 ESM strict의 named export 검출 한계 (gotcha #9 후보)

**증상**: `apps/api/src/routes/health.ts`에서 `import { pingDb, getWebhookPendingCount } from '@team-claude/db'` 실행 시 런타임 `SyntaxError: does not provide an export named 'getWebhookPendingCount'`. 다음 모든 시도 실패:
1. `import { pingDb, ... } from '@team-claude/db'` (root index.ts에 inline) → 실패
2. `import { pingDb, ... } from '@team-claude/db/queries'` (서브패스 + queries.ts 분리) → 실패
3. `import * as q from '@team-claude/db/queries'` (namespace) → import 성공하지만 `q.pingDb is not a function`

**원인 추정**: Node v24 ESM strict + tsx loader가 .ts 파일을 ESM로 처리하면서 named export 추론 실패. esbuild 변환 결과의 `__defProp` exports가 호스트 ESM 분석기와 호환 안 됨.

**해결**: `apps/api/src/lib/db.ts`에 raw better-sqlite3 connection 직접 작성 — packages/db 의존성 import 제거. 같은 dev.db 파일을 별도 connection으로 read (WAL이라 안전). gotcha #8 정합 — 4 PRAGMA 모두 set.

**향후**: packages/db build 추가(dist/ 사출) 시 정상 ESM named export 가능. 또는 dynamic import. 현재는 우회.

### 함정 ②: drizzle-orm 타입 hoisting 분리

**증상**: apps/api에서 `import { sql, count, eq } from 'drizzle-orm'` + `db.select(...).from(webhookJobs).where(eq(...))` 시 `'SQLiteColumn ...' is not assignable to parameter of type 'Column<...>'` (private property mismatch).

**원인**: pnpm hoisting이 drizzle-orm 0.38.4를 두 위치에 동일 instance 만들었으나 TypeScript가 두 .d.ts 파일을 분리 인스턴스로 인식. private property `shouldInlineParams`가 두 declaration 사이 비호환.

**해결**: drizzle-orm query builder 회피 — raw SQL prepared statement 사용 (lib/db.ts).

**향후**: pnpm.overrides로 단일 인스턴스 강제 또는 packages/db에 모든 query helper 캡슐화 후 build/dist 패턴.
